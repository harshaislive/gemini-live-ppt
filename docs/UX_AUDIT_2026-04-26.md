# Beforest Live Guide UX audit

Date: 2026-04-26

Scope: read-only audit of the Beforest Gemini Live presentation app for seamless audience experience. No app code was changed during this audit.

## Executive summary

The app is directionally strong: committed narration gives the core walkthrough a stable spine, Gemini Live is correctly treated as an interruption layer, and the black cinematic UI direction is closer to the intended premium feel.

The biggest issues are now operational and experience-continuity problems, not only copy or styling:

1. Subtitle timing is currently unreliable because declared narration durations no longer match regenerated WAV durations.
2. The first two beats explain Beforest more seriously, but the 10% access promise lands too late for a first-time listener.
3. Mic interruptions need a deterministic recovery model; several Live failure paths can still leave the audience confused or waiting.
4. Modal answers look interactive but do not actually route the fixed narration or CTA path.
5. The CSS has accumulated multiple override layers, which makes visual regressions likely.
6. Lead capture and conversion tracking are not production-grade yet.

## Priority 0: fix before any audience test

### 1. Subtitle metadata drift

Evidence:
- `presentationScript.ts` declares chunk durations such as `52` and `62` seconds for the first two chunks.
- The regenerated WAV files are materially shorter than those declarations.
- `buildTranscriptWindow()` uses declared `durationSeconds` for proportional subtitle timing.
- `ClientApp.tsx` drives subtitles from `audio.currentTime`, so bad duration metadata creates visible caption drift.

Impact:
- Viewers experience subtitles as late or disconnected from the speaker.
- This undermines perceived polish immediately.

Recommendation:
- Derive `durationSeconds` from actual WAV metadata during narration generation.
- Add a test that fails if declared duration and actual audio duration differ by more than 250ms.
- Medium-term: move to cue-based subtitles rather than proportional word slicing.

### 2. Full narration preload conflicts with video

Evidence:
- `layout.tsx` preloads every narration WAV in the document head.
- Narration alone is roughly 9MB; the active 720p video is roughly 7.6MB.

Impact:
- First-load bandwidth contention can create video stutter, especially on mobile and slower networks.

Recommendation:
- Preload only the first narration chunk.
- Prefetch the next chunk after playback starts.
- Convert delivery audio to AAC/Opus while keeping WAV as source artifacts only if needed.

## Priority 1: make the live experience deterministic

### 3. Mic error recovery is incomplete

Evidence:
- Active Live errors and send failures set unavailable states in multiple places.
- Some paths do not reliably call the same narrator-resume logic.

Impact:
- If Gemini Live drops during a mic turn, the narrator can remain paused or the user can feel stranded.

Recommendation:
- Treat every mic interaction as a bounded transaction.
- Add a single recovery routine for Live failures: stop recorder, stop Live playback, clear timers, close session, reset live refs, show a calm fallback, and resume narrator.
- Keep raw provider diagnostics out of the audience UI.

### 4. No escape hatch while Gemini is answering

Evidence:
- The primary button is disabled during `answering`.
- The pause/play control is hidden during live focus.

Impact:
- If an answer is slow, wrong, too long, or audio stalls, the audience cannot return to the walkthrough.

Recommendation:
- Add a visible `Return to walkthrough` or stop/skip action during answer playback.
- This action should stop Live audio, close the session, and resume committed narration deterministically.

### 5. Connecting state can disagree with narration state

Evidence:
- `livePhase` changes to `connecting` before mic/session are ready.
- Live-focus UI hides narration subtitles, but narration is paused only after both Gemini Live and mic access resolve.

Impact:
- On slow connect or permission delay, the UI says mic is preparing while narration may still be playing underneath.

Recommendation:
- Either pause narration immediately on mic intent and show a deliberate “opening mic” bridge, or keep normal narration UI visible until the mic is actually ready.
- Avoid hybrid states where audio and screen disagree.

### 6. Speech preview and server transcript need turn ownership

Evidence:
- Browser speech preview and Gemini input transcription both write into shared transcript refs.
- Late Gemini transcription can arrive after the user has tapped send or after narration resumes.

Impact:
- Text can appear late, duplicate, or contaminate the answer fallback.

Recommendation:
- Assign a `turnId` to each mic interaction.
- Keep local interim transcript separate from server-confirmed transcript.
- Freeze the displayed question when the user taps send.
- Ignore late Live messages once a turn is closed.

## Priority 2: improve narrative comprehension

### 7. The 10% promise lands too late

Evidence:
- Chunk 1 is now a restoration-first Beforest definition.
- The specific 10% promise and `thirty person-nights a year, for ten years` appear in chunk 2.

Impact:
- A first-time listener understands Beforest as restoration work before they understand why the presentation is relevant to them.

Recommendation:
- Keep Beforest-first framing, but bridge earlier:
  - Beforest is the restoration system.
  - 10% is the access path into that system without full ownership.
- Aim to land ecosystem + 30 person-nights within the first 45 seconds.

### 8. Ecosystem beat reads like a directory list

Evidence:
- The script lists beforest.co, Bewild, Hospitality, Experiences, and 10%.

Impact:
- Listeners may remember names but not the relationship between them.

Recommendation:
- Reframe as cause and effect:
  - The land is the core.
  - Food comes from it.
  - Hospitality lets you visit it.
  - Experiences take you deeper into it.
  - 10% lets you return every year without ownership.

### 9. Modal choices do not steer the walkthrough

Evidence:
- `handlePromptSubmit()` stores the selection, says `Noted`, and plays the next fixed chunk.

Impact:
- The interaction feels performative. The user answers, but nothing changes.

Recommendation:
- Either make modals honest pauses with no adaptive promise, or route each answer deterministically.
- At minimum, inject one tailored return line before the next chunk.
- For the final modal:
  - Trial answer should reveal/open trial CTA.
  - Updates answer should open updates flow.
  - Membership answer should route to structure recap.
  - Not-right-now should close softly.

## Priority 3: visual and interaction polish

### 10. CSS cascade is fragile

Evidence:
- Multiple late-stage CSS sections override earlier art direction layers.
- Brown glass, flat controls, editorial cleanup, and black interaction layers coexist.

Impact:
- Future edits can accidentally revive unwanted brown/green/card styles.

Recommendation:
- Consolidate CSS into:
  - tokens
  - base shell
  - video/scrim
  - narration UI
  - mic/live UI
  - modal/CTA UI
  - responsive rules
- Remove obsolete selectors for earlier prototypes.

### 11. Black overlay treatment is too uniform

Evidence:
- The same black surface treatment is applied to status, access card, CTA, modal, and live answer.

Impact:
- The app risks becoming flat black slabs over video instead of premium cinematic editorial.

Recommendation:
- Keep black strongest in the video scrim.
- Use more transparent near-black controls with hairline borders.
- Use sand only as a selective accent.
- Reduce heavy shadows on core controls.

### 12. Subtitles are too easy to truncate

Evidence:
- Subtitle CSS uses `white-space: nowrap`, overflow hidden, and ellipsis.

Impact:
- Important narration disappears mid-thought, especially on mobile.

Recommendation:
- Allow a controlled two-line subtitle area.
- Keep the subtitle container height stable to avoid layout jump.

### 13. CTA hierarchy is weak

Evidence:
- Final CTA shows trial stay and updates as near-equal actions.

Impact:
- The primary business action is not visually decisive.

Recommendation:
- Make `Take a trial stay` the dominant editorial action.
- Make updates a quieter secondary text action.
- Add a small destination cue for external navigation.

## Priority 4: conversion and launch readiness

### 14. Updates lead capture is not durable

Evidence:
- Contact details are stored in localStorage and passed in URL query params to Founding Silence.

Impact:
- Leads can be lost if popup open fails or the user closes the tab.
- PII appears in URLs/history/referrers.

Recommendation:
- Submit contact details to a real backend, CRM, sheet, or form endpoint before redirect.
- Use a lead id instead of PII query params.
- Add a success state and phone validation.

### 15. Error messages expose infrastructure details

Evidence:
- Token and Live failures can surface raw response text or socket messages.

Impact:
- Audience sees vendor/backend wording instead of calm recovery instructions.

Recommendation:
- Sanitize API errors.
- Log technical diagnostics server-side.
- Show user actions: `Retry mic`, `Continue without mic`, `Return to walkthrough`, `Contact Beforest`.

### 16. No conversion instrumentation

Evidence:
- Trial link and updates path are plain local UI actions.

Impact:
- Team cannot tell where users drop off or what they choose.

Recommendation:
- Track:
  - walkthrough started
  - first modal shown/answered
  - mic opened/sent/failed
  - final CTA shown
  - trial clicked
  - updates started/completed
  - external open failed or popup blocked

## Recommended execution sequence

### Phase 1: stability and trust

1. Fix actual audio duration metadata and subtitle drift.
2. Remove all-upfront WAV preload.
3. Add unified Live failure recovery.
4. Add `Return to walkthrough` during answer state.
5. Sanitize user-facing Live/token errors.

### Phase 2: flow clarity

1. Compress and reframe first two beats so Beforest and 10% both land within 45 seconds.
2. Make ecosystem explanation causal, not directory-like.
3. Make modal choices deterministic or reduce their adaptive promise.
4. Make final modal route directly to trial, updates, membership recap, or close.

### Phase 3: premium UI consolidation

1. Consolidate CSS and remove dead prototype layers.
2. Tune black overlay system so video stays cinematic.
3. Make subtitles two-line and stable.
4. Strengthen CTA hierarchy.
5. Convert mobile modals to bottom-sheet style.

### Phase 4: production launch readiness

1. Add durable lead capture.
2. Add analytics events.
3. Add Playwright smoke tests with mobile viewport and throttled network.
4. Add dropped-frame and subtitle-drift checks.
5. Extend QA checklist with conversion and failure-path tests.

## Team test focus

Before the next audience demo, the team should specifically test:

- Does a first-time viewer understand Beforest and 10% within 45 seconds?
- Does the app always say `30 person-nights`, never only `30 nights`?
- Do subtitles match the voice on every narration chunk?
- Does the mic recover if Gemini Live fails mid-question?
- Can the user return to narration while an answer is slow?
- Do modal answers actually change the next action?
- Is the final trial-stay CTA clearly primary?
- Are updates leads received somewhere durable?
- Does mobile video remain smooth on a throttled network?
- Does any raw vendor/backend error text appear?
